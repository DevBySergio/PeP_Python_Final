1. Contexto

Qué problema resuelve el proyecto

Objetivo del sistema

Automatizar el procesamiento de ficheros de clientes y tarjetas, aplicar limpieza y validación de calidad, proteger la información sensible y dejar los datos listos para consulta en PostgreSQL y para auditoría en CSV de salida.

El pipeline acepta ficheros con patrón de nombre controlado y procesa solo los que cumplen la convención esperada.

Valor de negocio

  • Reduce trabajo manual y errores operativos en la carga de datos.
  • Estandariza la validación de DNI, teléfono y correo.
  • Protege datos personales y financieros mediante hashing con SALT.
  • Permite ejecución programada diaria y repetible desde Airflow.
2. Arquitectura

Arquitectura funcional de extremo a extremo

El diseño separa claramente entrada de datos, lógica de transformación, persistencia y orquestación, lo que hace el proyecto más mantenible y fácil de escalar.

1. Entrada

CSV en data/input con nombres como Clientes-YYYY-MM-DD.csv y Tarjetas-YYYY-MM-DD.csv.

2. Extract

El módulo etl/extractor.py descubre archivos válidos y los carga con pandas.

3. Transform

Se limpian campos, se validan reglas de negocio y se anonimizan valores sensibles.

4. Load

Los datos depurados se guardan en CSV de salida y se insertan en tablas PostgreSQL.

5. Orquestación

Airflow ejecuta el proceso cada día a las 03:00 y permite trazabilidad por logs.

3. Stack

Tecnologías utilizadas y por qué se eligieron

Python 3.11

Lenguaje principal del ETL por su productividad, claridad y ecosistema de ingeniería de datos.

Pandas

Permite leer CSV, transformar DataFrames y exportar resultados de forma muy eficiente.

SQLAlchemy + psycopg2

Gestionan la conexión a PostgreSQL y facilitan la carga de datos.

Apache Airflow

Orquesta la ejecución diaria, centraliza el control y deja visible el historial del pipeline.

PostgreSQL 15

Base de datos relacional robusta, adecuada para persistir el resultado estructurado del ETL.

Docker y Compose

Garantizan un despliegue reproducible y sencillo, con todos los servicios del entorno coordinados.

.env

Centraliza configuración sensible y evita hardcodear credenciales o SALTs en el código.

SHA-256 + SALT

Se usa para anonimización irreversible de DNI, tarjeta y CVV antes de su almacenamiento.

5. Flujo ETL

Qué ocurre exactamente durante el procesamiento

Extract

Descubrimiento y lectura

Se listan los ficheros de entrada y solo se aceptan nombres con formato fechado correcto.

Transform

Limpieza de datos

Se eliminan espacios, se normalizan nombres y correos y se limpia el teléfono dejando solo dígitos.

Validate

Reglas de calidad

El DNI se valida con algoritmo real de letra, y además se comprueban correo y teléfono.

Flagging

Trazabilidad del dato

Cada validación genera pares de banderas ok/ko con valores Y/N.

Protect

Anonimización

Los campos sensibles no se guardan en claro: se convierten a hash SHA-256 concatenando un SALT.

Load

Persistencia

Los resultados se exportan a data/output y se cargan en tablas PostgreSQL.

6. Seguridad

Protección de datos y control de calidad

Anonimización aplicada

  • dni se transforma en dni_hash.
  • numero_tarjeta se transforma en numero_tarjeta_hash.
  • cvv se transforma en cvv_hash.
  • La SALT se obtiene desde variables de entorno, no desde código hardcodeado.

Garantías del diseño

  • Los CSV de salida ya contienen la versión protegida de los datos críticos.
  • Se preserva utilidad analítica sin exponer información identificable.
  • El uso de flags facilita auditoría y revisión de calidad por registro.
  • La conexión a base de datos también se parametriza con entorno. 
def hash_value(value: str) -> str:
    raw = f"{HASH_SALT}{value}".encode("utf-8")
    return hashlib.sha256(raw).hexdigest()
7. Despliegue

Cómo se despliega y ejecuta el sistema

Contenedores

El proyecto levanta PostgreSQL, un paso de inicialización de Airflow, el webserver y el scheduler.

Imagen base

El Dockerfile extiende apache/airflow:2.8.1 e instala dependencias del proyecto.

Variables

Compose inyecta credenciales de PostgreSQL, host, puerto y HASH_SALT al entorno.

8. Operación

Qué se puede mostrar en una demo real

Secuencia recomendada de demostración

  1. Mostrar los CSV en data/input.
  2. Abrir Airflow y localizar el DAG etl_pipeline.
  3. Lanzar un Manual Run o explicar la programación diaria a las 03:00.
  4. Enseñar los CSV generados en data/output.
  5. Consultar las tablas clientes y tarjetas en PostgreSQL.

Datos de la demo incluida

  • 2 archivos de entrada válidos en la carpeta de demo.
  • 3 registros de clientes procesados.
  • 3 registros de tarjetas procesados.
  • Todos los datos sensibles aparecen ya anonimizados en salida.
9. Diseño técnico

Fortalezas, decisiones y posibles mejoras

Fortalezas actuales

  • Separación modular por responsabilidades.
  • Pipeline reutilizable tanto manualmente como desde Airflow.
  • Esquema de datos claro y fácil de extender.
  • Protección de datos integrada en la propia transformación.
  • Entorno reproducible gracias a Docker Compose.

Mejoras futuras razonables

  • Tests automatizados unitarios y de integración.
  • Control más estricto de duplicados y versionado de cargas.
  • Métricas de calidad y reporting en Airflow.
  • Manejo avanzado de errores y alertas operativas.
  • Posible evolución a almacenamiento cloud o pipeline incremental.
10. Cierre

Conclusión

El proyecto demuestra un flujo ETL completo y profesional: procesamiento estructurado, transformación con reglas de negocio, anonimización de datos sensibles, carga automatizada y despliegue reproducible en contenedores.

Más allá de ser una práctica técnica, enseña cómo diseñar una solución de datos ordenada, segura y preparada para operar de forma periódica.
